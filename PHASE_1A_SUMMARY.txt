NOTICED / FLAGS
--------------------------------------------------------------------------------
1. OPEN ACTION (verbatim from the spec — addressed to the user, not the
   assistant): Before deploying the new public-facing frontend, rotate the
   Neon DB password (exposed in chat history multiple times) and the
   JWT_SECRET (currently weak: `luz2222`). Do this as a separate small
   commit — update Vercel env vars, regenerate Neon credentials, redeploy.
   This invalidates all active JWTs (staff + member), which is intentional.

2. helmet defaults add a Strict-Transport-Security header. That is benign
   over HTTPS (Vercel terminates TLS) but means anyone hitting the API
   directly over HTTP locally will see a one-shot HSTS upgrade attempt in
   their browser. Harmless for curl/postman; flagging for any future
   browser-based local testing.

3. The honeypot field is `website`. If the eventual landing-page form ever
   needs a legitimate "Your website" field, change the honeypot name
   first — or add a second hidden field with a name like `url_homepage`
   and switch the controller check.

4. The captcha hook in submitContact is commented and pre-positioned BEFORE
   the honeypot check so failed captchas produce a clear 400 (not a silent
   201). Add a CAPTCHA_ENABLED env var + verifyCaptcha helper when wiring.

5. `lib/validation/leads.js` accepts `website` in the Zod schema (rather
   than rejecting unknown keys) so bots receive a 201 instead of a clear
   "invalid field" error and don't learn to adapt. If you later switch the
   Zod schemas to strict mode somewhere, the honeypot must NOT be made
   strict.

6. helmet ^8 is the latest major as of writing. If you bump it later,
   re-verify CSP defaults — v9 may flip a flag.

7. contact_submission.assigned_to has ON DELETE SET NULL. Deleting a
   `personal` row will null out the assignment but leave the lead. That
   matches the data-retention intent (leads are a record), but staff
   queries that GROUP BY assigned_to should remember to handle NULL.

8. The 404 handler in server.js still calls helmet-stamped responses — fine
   — but it doesn't pass through the errorHandler. Out of scope; flagging.
